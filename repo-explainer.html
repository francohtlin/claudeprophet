How this forecasting harness works

This repository is a standalone Claude-native Prophet Hacks forecasting harness. It launches Claude with forecasting instructions, local domain skills, lookup CLIs, and a JSON validator so each task ends as a calibrated probability distribution instead of an informal answer.

Plain-English summary

Node scripts are the front door, Bash launchers start Claude, `AGENTS.md` defines the agent contract, Python modules gather evidence, and `submit_prediction` verifies the final forecast shape.

Not DockerizedRuns directly in the local repo with the local Claude subscription session (`claude login`).
Default Model`claude-opus-4-8`, overrideable with `CLAUDE_FORECAST_MODEL`.
SearchClaude has the built-in WebSearch tool, so it can use live web evidence.
OutputFinal forecasts are JSON with exact event outcomes and probabilities.

The full forecast loop

The repository is organized around one repeatable agent loop: parse the event, gather evidence, calibrate, validate, and return strict JSON.

1

User input

The user passes either plain task text or a path to an event JSON file through `npm run claude:prophet` or `npm run claude:prophet:exec`.

2

Prompt assembly

The launcher reads `prompts/goal_prompt.md`, appends the event or task, and treats it as the Claude goal.

3

Agent contract

Claude starts in the repo root, reads `AGENTS.md`, and follows the required forecasting loop and output shape.

4

Skill selection

`skills/calibration-validation` is mandatory. Domain skills are selected from finance, sports, macro, politics, culture, sociology, and market discovery.

5

Evidence lookup

The agent uses local CLIs first, then web search for official sources, current news, rulebooks, odds, filings, calendars, and polls.

6

Validation

The final JSON is checked with `submit_prediction`; failures are revised until labels, probabilities, and rationale conform.

Launch path

The entrypoints are npm scripts, but the real launch logic lives in two Bash scripts.

Interactive run

`npm run claude:prophet -- ...` calls `scripts/launch_goal.sh`. It starts an interactive Claude (Claude Code) session with the working directory set to this repo, the built-in WebSearch tool available, and a permission mode that allows workspace edits without approval prompts by default.

npm run claude:prophet -- sample_events/sample-economics.json
npm run claude:prophet -- "Will Bitcoin trade above 100000 by Sunday?"

One-shot run

`npm run claude:prophet:exec -- ...` calls `scripts/run_goal_exec.sh`. It uses `claude --print`, and the final answer must be strict JSON in the `schemas/final_forecast.schema.json` shape, enforced by the prompt instructions plus the harness validator/fallback.

npm run claude:prophet:exec -- sample_events/sample-economics.json

Command surface

These scripts are the repo's supported tools. They are intentionally called through npm so the virtualenv path stays consistent.

npm run setup
Creates `.venv`, upgrades pip, and installs `requirements.txt`.
npm run check
Checks Claude login status, then runs the unittest-based lookup tests and the pytest validator tests.
npm run market:lookup
Searches normalized prediction-market records across Polymarket Gamma and Kalshi. PMXT is disabled unless `MARKET_LOOKUP_ENABLE_PMXT=1` is set.
npm run kalshi:discover
Retrieves Kalshi markets by status, series ticker, event ticker, category, and text query. It can optionally include orderbook depth.
npm run sports:lookup
Uses ESPN scoreboard data by default and optional bookmaker odds from The Odds API when `THE_ODDS_API_KEY` is configured.
npm run finance:lookup
Routes symbols and queries across equity, crypto, macro, news, and filings providers such as yfinance, CCXT, FRED, Finnhub, Alpha Vantage, and SEC tooling when installed or configured.
npm run submit:prediction
Runs local validation only. It never sends an external submission. Use `--require-probability-sum` for strict normalization.

What each module does

The Python packages normalize external evidence into agent-readable JSON. They do not forecast by themselves.

market_lookup

General market discovery

Combines provider search results, deduplicates markets, annotates whether close time is before the event's ideal close time, and orders records by status, timing fit, liquidity, and source diversity.

  • Polymarket Gamma provider
  • Kalshi provider
  • Optional PMXT provider
kalshi_discovery

Kalshi-specific scan

Pages through Kalshi markets with filters for market status, event ticker, series ticker, or category. Query matching is applied after retrieval using meaningful terms.

  • Optional raw output
  • Optional orderbooks
  • Large max-market caps for broad scans
sports_lookup

Schedules, scores, odds

Infers sport from the query or accepts an explicit sport, pulls ESPN events over a date window, and optionally adds U.S. bookmaker markets from The Odds API.

  • NFL, NBA, MLB, NHL
  • Soccer and tennis support
  • Self-test command included
finance_lookup

Finance and macro data

Parses tickers, routes symbols as equity or crypto, fetches prices/history/news, and can add macro series or filings depending on requested data and environment configuration.

  • Yahoo Finance for equities
  • CCXT for crypto
  • FRED and SEC support when configured
submit_prediction

Schema and rule guard

Checks that forecasts are JSON objects with `probabilities` and `rationale`, verifies exact outcome labels against event JSON, rejects duplicate or extra labels, and optionally requires probabilities to sum to 1.

  • Accepts legacy `p_yes` for binary events
  • Normalizes valid forecasts
  • Reports errors and warnings as JSON
sample_tools

Resolved-event helper

`sample_tools.build_actuals` converts resolved sample events into actuals artifacts for scoring or local evaluation workflows.

  • Works with `sample_events/sample-resolved.json`
  • Writes `sample-resolved-actuals.json`
skills

Agent-side forecasting playbooks

Skills are concise domain instructions. The agent reads only relevant ones per event, plus calibration-validation every time.

  • Domain evidence checklists
  • Calibration notes
  • Tool command hints
tests

Behavior locks

Tests cover provider normalization, lookup behavior, Kalshi discovery, sports and finance self-test surfaces, and strict prediction validation.

  • `unittest` for lookup modules
  • `pytest` for validator cases
  • Run through `npm run check`

Event and forecast shapes

The event file controls the forecast labels. The prediction must mirror those labels exactly.

Input event shape
{
  "event_ticker": "KX...",
  "market_ticker": "KX...",
  "title": "What will happen?",
  "description": "Resolution context...",
  "category": "Economics",
  "rules": "Exact resolution rules...",
  "close_time": "2026-06-10T12:29:00Z",
  "outcomes": ["Outcome A", "Outcome B"],
  "resolved_outcome": null
}
Required final forecast shape
{
  "probabilities": [
    {"market": "Outcome A", "probability": 0.57},
    {"market": "Outcome B", "probability": 0.43}
  ],
  "rationale": "Brief evidence and calibration note."
}

Validation rules

`submit_prediction` is the guardrail that turns a forecast into a valid hackathon response.

Labels must match

For multi-outcome events, every event outcome must appear exactly once in `probabilities`. Missing labels, invented labels, and duplicate labels fail validation.

Probabilities must be finite

Each probability must be a number from 0 to 1. With `--require-probability-sum`, the total must equal 1 within a small tolerance.

Rationale is required

The validator requires a non-empty rationale unless `--no-require-rationale` is explicitly passed. Extra top-level keys are ignored with warnings.

mkdir -p tmp
npm run submit:prediction -- \
  --event tmp/event.json \
  --prediction tmp/prediction.json \
  --require-probability-sum

Setup and provider configuration

The repo works in tiers: core Python validation is local, while richer data depends on installed packages and optional API keys.

Local bootstrap

Run setup once from the repo root. It builds the Python virtualenv and installs the default requirements.

cd /path/to/claude-prophet
npm run setup
npm run check

Optional environment

Provider keys are read from the local environment. The repo includes `.env.example`; `local_env.py` loads local values for tools that call it.

  • `THE_ODDS_API_KEY` for bookmaker odds
  • `FRED_API_KEY` for macro series
  • `FINNHUB_API_KEY` or `ALPHA_VANTAGE_API_KEY` for finance extras
  • `SEC_USER_AGENT` or `SEC_COMPANY_NAME` plus `SEC_EMAIL` for SEC tooling
  • `MARKET_LOOKUP_ENABLE_PMXT=1` to enable PMXT market lookup

Important boundaries

These details matter when debugging or extending the harness.

The validator does not submit externally

`submit_prediction` says whether the JSON conforms to the Prophet Hacks shape. It returns a validation report and normalized payload, but performs no remote submission.

Market data is evidence, not the answer

`AGENTS.md` requires the agent to compare market-implied probabilities with rules and evidence, and to discount stale, illiquid, or rule-mismatched markets.

Where to change things

Use the repo boundaries below to avoid editing the wrong layer.

Change agent behavior
Edit `AGENTS.md` for the durable forecasting contract, or `skills/*/SKILL.md` for domain-specific evidence and calibration playbooks.
Change launch defaults
Edit `scripts/launch_goal.sh` or `scripts/run_goal_exec.sh`, or override runtime values with `CLAUDE_FORECAST_MODEL`, `CLAUDE_FORECAST_PERMISSION_MODE`, and `CLAUDE_FORECAST_APPROVAL`.
Add a data provider
Add provider code under the relevant lookup package, normalize output to existing JSON style, then extend tests under `tests/`.
Tighten output validation
Update `submit_prediction/validator.py`, keep `schemas/final_forecast.schema.json` aligned for `claude --print`, and add validator tests.
Add sample coverage
Use the `sample:retrieve:*` scripts to fetch Prophet sample datasets, then run tests or sample actuals generation as needed.
